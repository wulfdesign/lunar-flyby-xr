🚀 Artemis Orbital Physics & Math Reference

This "Live Documentation" explicitly catalogs the mathematics driving the WebXR simulation. It is decoupled from the main codebase to keep the JavaScript highly optimized for 7,200x time-warp integration loops, while still providing deep educational context.

1. Velocity Verlet Integration

Unlike standard visualizations that place ships on predefined "rails" (splines), this engine calculates raw gravity frame-by-frame. We use Velocity Verlet instead of Euler integration because it handles multi-day orbital paths with significantly less "energy drift" (preventing the orbit from artificially spiraling outward over time).

2. N-Body Gravity Constants

All values strictly follow standard real-world dimensions: Kilometers (km), Kilograms (kg), and Seconds (s).

  • radEarth = 6371 -- Earth's equatorial radius (km)
  • radMoon = 1737 -- Lunar equatorial radius (km)
  • distMoon = 384400 -- Average Earth-to-Moon distance (km)
  • muEarth = 398600.4418 -- Earth's Standard Gravitational Parameter (G × M_Earth) in km³/s²
  • muMoon = 4902.8000 -- Moon's Standard Gravitational Parameter in km³/s²
Calculated Gravity Acceleration Vector:

Δv = - (μ / |r|³) * r

3. Atmospheric Re-Entry & Aerodynamic Drag

To survive, the flight computer must thread a brutal 2-degree reentry keyhole. If the Entry Angle is steeper than -7.0°, lethal deceleration G-forces will destroy the structural integrity. If shallower than -5.5°, aerodynamic drag won't scrub your velocity, skipping the craft off the atmosphere into solar orbit.

The Barometric Formula (Exponential Density):

  • RHO_0 = 1.225 -- Sea level atmospheric density (kg/m³)
  • SCALE_HEIGHT = 8.5 -- Earth's atmospheric scale height (km)
  • DRAG_AREA = 0.011 -- Exposed cross-sectional area of the Orion capsule base (km²)
Atmospheric Density Equation:
ρ = ρ0 * e^(-altitude / H_scale)

Drag Force Equation:
F_d = 0.5 * ρ * v² * C_d * A

4. Performance: Zero-Allocation Object Tracking

At a maximum 7,200x time warp, allocating new THREE.Vector3() every frame would trigger massive JavaScript Garbage Collection pauses and destroy the frame rate. Instead, we globally declare _tempVec object pools and continually execute .copy() and .add() into them, allocating absolutely zero new memory during the multi-day flight loop.
